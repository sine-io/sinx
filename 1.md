
# Refactoring SinX with DDD Lite + CQRS + Clean Architecture + DIP

Based on my analysis of the SinX codebase, here's a comprehensive refactoring strategy to implement DDD Lite + CQRS + Clean Architecture + DIP principles.

## Current Architecture Analysis

SinX is currently structured as a monolithic application where the `Agent` struct serves as the central component managing all functionality. The main entry point delegates to CLI commands, and core functionality is concentrated in the `dkron` package.

## Domain Identification (DDD Lite)

### Bounded Contexts

1. **Job Management**: Job creation, validation, and lifecycle
2. **Scheduling**: Cron-based job scheduling and triggering  
3. **Execution**: Job execution and result processing
4. **Cluster Management**: Node discovery, leadership, and coordination
5. **Storage**: Persistent state management

### Domain Entities

- **Job**: Core scheduling unit
- **Execution**: Job execution instance
- **Agent**: Cluster node representation

## Proposed Clean Architecture Structure

```text
cmd/
├── api/           # API server command
├── agent/         # Agent command  
└── cli/           # CLI utilities
│
internal/
├── domain/        # Domain layer (entities, value objects, domain services)
│   ├── job/
│   ├── execution/ 
│   ├── scheduler/ 
│   ├── cluster/
│   └── shared/
├── application/   # Application layer (use cases, CQRS handlers)
│   ├── commands/
│   ├── queries/
│   └── handlers/
├── infrastructure/ # Infrastructure layer (external concerns)
│   ├── storage/
│   ├── messaging/
│   ├── scheduling/
│   └── network/
├── interfaces/    # Interface adapters (HTTP, gRPC, CLI)
│   ├── http/
│   ├── grpc/
│   └── cli/
└── main.go
```

## CQRS Implementation Strategy

### Commands (Write Operations)

- `CreateJobCommand`
- `UpdateJobCommand`
- `DeleteJobCommand`
- `TriggerJobCommand`
- `ToggleJobCommand`

### Queries (Read Operations)

- `GetJobQuery`
- `ListJobsQuery`
- `GetExecutionQuery`
- `ListExecutionsQuery`
- `GetClusterStatusQuery`

### Command/Query Handlers

Separate handlers for each command and query, implementing the mediator pattern for decoupling.

## Step-by-Step Refactoring Plan

### Phase 1: Extract Domain Layer

1. **Create Domain Entities**:
   - Extract `Job` entity from current implementation
   - Extract `Execution` entity
   - Define domain interfaces (repositories, services)

2. **Define Value Objects**:
   - `Schedule` (cron expression validation)
   - `ExecutionStatus`
   - `NodeID`
   - `JobConfig`

3. **Create Domain Services**:
   - `JobValidationService`
   - `SchedulingService`
   - `ExecutionService`

### Phase 2: Implement Application Layer

1. **Create Command Handlers**:

   ```text
   internal/application/commands/
   ├── create_job_handler.go
   ├── update_job_handler.go
   ├── delete_job_handler.go
   └── trigger_job_handler.go
   ```

2. **Create Query Handlers**:

   ```text
   internal/application/queries/
   ├── get_job_handler.go
   ├── list_jobs_handler.go
   └── get_execution_handler.go
   ```

3. **Implement Command/Query Bus**:
   - Central dispatcher for CQRS operations
   - Request/response handling with proper error management

### Phase 3: Infrastructure Layer Refactoring

1. **Extract Storage Layer**:
   - Abstract current BuntDB implementation
   - Create repository interfaces and implementations
   - Implement Unit of Work pattern

2. **Extract Scheduling Infrastructure**:
   - Separate cron scheduling logic
   - Create scheduler interface and implementation
   - Handle job triggering and execution coordination

3. **Extract Network Layer**:
   - Abstract gRPC communication
   - Abstract Serf cluster management
   - Create interfaces for cluster communication

### Phase 4: Interface Adapters

1. **HTTP API Layer**:
   - Extract current API handlers
   - Implement REST endpoints using command/query handlers
   - Add proper request/response DTOs

2. **gRPC Interface**:
   - Refactor current gRPC implementation
   - Map gRPC calls to command/query handlers

3. **CLI Interface**:
   - Refactor CLI commands
   - Use application layer handlers instead of direct agent access

### Phase 5: Dependency Injection (DIP)

1. **Create Dependency Container**:
   - Use Go's interface-based DI or a lightweight DI container
   - Define service interfaces and wire implementations

2. **Configuration Management**:
   - Extract configuration logic
   - Create configuration interfaces and implementations

3. **Plugin System Integration**:
   - Refactor plugin interfaces
   - Integrate with dependency injection system

## Implementation Guidelines

### Domain Layer Rules

- No external dependencies (database, HTTP, etc.)
- Pure business logic only
- Interfaces define contracts with outer layers

### Application Layer Rules  

- Orchestrates domain objects
- Contains use case implementations
- Depends only on domain layer

### Infrastructure Layer Rules

- Implements domain interfaces
- Contains external system integrations
- Database, messaging, external APIs

### Interface Layer Rules

- Translates external requests to application commands/queries
- Handles serialization/deserialization
- No business logic

## Migration Strategy

1. **Gradual Refactoring**: Start with one bounded context (Job Management)
2. **Feature Toggles**: Use flags to switch between old and new implementations
3. **API Compatibility**: Maintain existing API contracts during transition
4. **Testing**: Comprehensive unit and integration tests for each layer
5. **Documentation**: Update architectural documentation as you refactor

## Benefits of This Approach

1. **Separation of Concerns**: Clear boundaries between business logic and technical concerns
2. **Testability**: Each layer can be tested independently
3. **Maintainability**: Changes in one layer don't affect others
4. **Scalability**: Different layers can be scaled independently
5. **Plugin Extensibility**: Better plugin architecture with clear interfaces

## Notes

This refactoring will significantly improve the codebase's maintainability and extensibility while preserving SinX's core functionality. The current monolithic `Agent` struct will be decomposed into focused, single-responsibility components following Clean Architecture principles. The plugin system will integrate seamlessly with the new dependency injection approach, making it easier to extend SinX's capabilities.

Start with Phase 1 and gradually migrate functionality while maintaining backward compatibility. This approach ensures minimal disruption to existing users while modernizing the architecture.
